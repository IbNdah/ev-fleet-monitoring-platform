# 🏗️ Architecture Guide

This document explains the architectural design of the **EV Fleet Monitoring Platform**.

Its purpose is to provide a concise overview of the platform, explain the main design decisions, and help readers understand how telemetry flows through the system.

---

# Architecture at a Glance

The platform is built around a modern **event-driven serverless architecture** on Microsoft Azure.

Telemetry generated by simulated electric vehicles is securely ingested, processed, stored, and exposed through REST APIs before being visualized in Azure Managed Grafana.

The architecture emphasizes:

- Scalability
- Maintainability
- Security
- Observability
- Infrastructure as Code

# Architecture Layers

The solution is organized into seven logical layers.

| Layer | Responsibility |
|--------|----------------|
| Device | Generates EV telemetry |
| Edge | Collects and forwards telemetry |
| Ingestion | Receives cloud events |
| Processing | Processes and enriches telemetry |
| Data | Stores telemetry |
| Service | Exposes REST APIs |
| Visualization | Displays operational dashboards |

# Key Architectural Decisions

## 1. Event-Driven Processing

Telemetry is processed asynchronously through Azure IoT Hub and Azure Event Hub.

This design decouples producers from consumers, improves scalability, and enables the platform to handle increasing telemetry volumes without changing the overall architecture.

---

## 2. Serverless Computing

Azure Functions were selected to minimize operational overhead while providing automatic scaling.

The platform processes telemetry only when events occur, reducing infrastructure management and aligning with cloud-native design principles.

---

## 3. Cosmos DB for Telemetry Storage

Azure Cosmos DB provides low-latency NoSQL storage optimized for high-frequency telemetry ingestion.

Its flexible schema and scalability make it well suited for time-series inspired IoT workloads.
